## Выполненные изменения

### Бэкенд — рефакторинг архитектуры

Монолитные `service.py` и `tasks.py` разбиты на слои:

| Слой | Назначение |
|------|------------|
| `api/routes/` | HTTP-эндпоинты |
| `api/deps.py` | DI через FastAPI Depends |
| `services/` | Бизнес-логика |
| `repositories/` | Доступ к данным |
| `models/` | SQLAlchemy-модели |
| `db/session.py` | Единый engine и сессии |
| `core/config.py` | Настройки через `pydantic-settings` |
| `tasks/` | Celery-воркер и пайплайн обработки |

Публичный API и бизнес-логика сохранены: те же эндпоинты, эвристики сканирования, извлечение метаданных и алерты.

### Бэкенд — оптимизации

1. **Объединение Celery-пайплайна** — три задачи (`scan` → `extract_metadata` → `send_alert`) заменены одной `process_file` в рамках одной DB-сессии. Меньше round-trip'ов в Redis и повторных SELECT по одному файлу.
2. **Потоковая загрузка файлов** — запись на диск чанками по 1 MiB вместо чтения всего файла в память.
3. **Потоковое извлечение метаданных** — текстовые файлы читаются построчно, PDF сканируется чанками.
4. **Единый пул подключений к БД** — один `engine` для API и Celery-воркера вместо дублирования в разных модулях.

### Фронтенд — разбиение на слои

Логика вынесена из `app/page.tsx` в слои по аналогии с FSD:

- `entities/` — типы доменных сущностей
- `features/` — сценарии (`useFileDashboard`, `useUploadFile`, `UploadFileModal`)
- `shared/api/` — HTTP-клиент и методы API
- `shared/lib/` — утилиты (форматирование, бейджи)
- `widgets/` — `FileTable`, `AlertList`

Страница `app/page.tsx` осталась тонким слоем композиции.
